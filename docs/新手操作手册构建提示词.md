# 新手操作手册构建提示词

> **文档职责**：沉淀一份可复用的提示词，用于为其他项目生成面向新读者的操作手册。
> **适用场景**：你已经拿到一个新项目的代码、文档、产品页面或原型，希望让 AI 帮你输出一份可维护、可执行、对新手友好的操作手册。
> **阅读目标**：读完后，你应当能够把这份提示词直接交给 AI，并稳定生成结构清晰、步骤明确的新用户操作文档。

## 使用方式

这份提示词适合在你已经准备好下面信息后使用：

- 项目代码仓库或主要源码目录
- README、产品文档、设计稿、原型图或页面截图
- 真实可操作的前端页面或产品后台
- 你希望面向的读者类型
- 你希望文档覆盖的核心流程

如果信息不完整，也可以先用这份提示词让 AI 输出第一版，再逐步补充。

## 推荐输入材料

把下面内容尽量提供给 AI：

1. 项目简介
2. 目标读者
3. 产品或平台的主要页面
4. 实际操作流程
5. 关键依赖关系
6. 已知坑点或常见误解
7. 你喜欢的文风或排版偏好

## 通用提示词

下面这段可以直接复制使用。

```md
你现在要为一个项目编写一份“新手操作手册”，目标读者是第一次接触该项目的新用户。

你的任务不是写功能说明书，也不是写开发文档，而是写一份真正能带着新手一步步操作的用户手册。

请严格遵守下面要求：

## 一、写作目标

这份文档要解决的问题是：

- 新用户打开系统后，不知道先做什么
- 界面上虽然有很多按钮，但缺少操作顺序
- 不知道为什么这样操作
- 不知道什么时候算做对了
- 不知道哪些步骤有依赖关系

所以文档重点必须是：

1. 操作顺序
2. 每一步为什么这么做
3. 做完后看什么来判断是否正确
4. 哪些地方容易误解
5. 什么时候可以进入下一步

## 二、你必须先做的事

在写文档前，你必须先分析以下内容：

1. 代码结构
2. 现有 README / 文档
3. 前端页面结构或后台界面
4. 实际按钮、表单、列表、弹窗、工作流
5. 各步骤之间的依赖关系
6. 哪些内容是主流程，哪些只是辅助信息

如果你没有足够依据，请不要臆测。应根据代码、界面和现有文档综合判断。

额外要求：

- 优先相信源码和真实页面，其次才是 README 或历史说明
- 按页面里的真实按钮名、区块名、弹窗名来写，不要擅自改写成交互上并不存在的名字
- 如果某个功能在界面上“看起来像能用”，但源码显示只是占位或 Beta，请按真实状态写
- 如果某个功能已经可用，就不要因为旧文档里写过 `Beta` 而继续照抄

## 三、文档定位

这份文档是“新手操作手册”，不是：

- 功能罗列文档
- 参数百科
- 技术架构说明
- 二次开发说明

如果某个界面元素不会影响主流程，请轻写，不要喧宾夺主。

## 四、文档风格要求

整体风格要求：

- 面向新手
- 步骤优先
- 解释克制
- 排版清晰
- 不要大段纯文字堆砌
- 不要写成培训课件
- 不要写成产品宣传稿

表达要求：

- 先告诉用户做什么，再简短解释为什么
- 每一步都要可执行
- 避免空泛表述
- 避免“建议用户自行探索”
- 语气要稳，不要过度热情，不要口号化

可以有少量 emoji，但只能起到辅助分隔作用，不能破坏专业感。

## 五、文档整体结构

文档开头固定使用：

> **文档职责**：
> **适用场景**：
> **阅读目标**：

正文推荐结构：

1. 开始前确认
2. 先理解整体生产链路 / 主流程
3. 术语速查表
4. 第一次练习建议
5. 主流程步骤
6. 常见问题
7. 一次练习后你应该掌握什么

## 六、每个主步骤的固定结构

每个步骤都必须使用统一模板：

### X.0 界面结构
用表格呈现，列固定为：
- 区域
- 什么时候用
- 作用

如果某一步界面明显分成两层，也允许拆成两张小表，例如：

- 页面级区域
- 详情页 / 弹窗 / 工作台

### X.1 操作顺序
这是整章主干，必须最重要。

写法要求：
- 使用“步骤 1 / 步骤 2 / 步骤 3”这样的子标题
- 每个步骤下面固定写：
  - 操作
  - 原因
  - 结果
- 必要时可以补一行“建议”或“重点检查”

如果某一步是参数较多、模式较多、按钮较多的复杂页面，可以在 `X.1 操作顺序` 内穿插少量辅助表格，但这些表格只能服务操作顺序，不能反过来喧宾夺主。

### X.2 判断标准
回答：
- 什么状态说明这一步基本做对了
- 什么情况下可以进入下一步
- 哪些结果是正常现象，不必误判成错误

### X.3 关键说明
这里放：
- 依赖关系
- 不会自动联动的地方
- 容易误解的点
- 需要回退重做的条件

每个步骤结尾必须有：

**完成标志**：

一句话说明什么状态下可以进入下一步。

## 七、操作顺序写法要求

操作顺序必须优先于界面解释。

你要像带新手上手一样写：

- 先点什么
- 再看什么
- 不对时改什么
- 改完后再做什么

不要只解释“这个按钮是什么”，而要告诉读者“什么时候点这个按钮”。

如果某一步存在先后依赖，必须明确写出来，例如：

- 先生成静态，再生成动态
- 先定主参考图，再做派生参考图
- 先看顺序，再生成图片
- 先选版本，再执行拼接

如果某一步是“先选模式，再走不同分支”，必须明确分支，例如：

- 如果用 `I2V`，先做 A，再做 B
- 如果用 `R2V`，先做 C，再做 D

## 八、主次关系要求

你必须主动判断哪些内容是：

- 主流程核心步骤
- 辅助信息
- 可选项
- Beta 能力
- 非第一次练习重点

处理原则：

- 主流程写重
- 辅助信息写轻
- 可选项明确写“不是必做”
- Beta 能力不要抢主流程篇幅

同时要避免另一个错误：

- 不要因为某页是后续能力，就一概写成“先跳过”
- 如果源码显示它已经能完成真实操作，就应该写清楚“现在能做什么”
- 如果它只是部分可用，也要写清楚“当前可用部分”和“当前边界”

## 九、排版要求

必须注意阅读观感：

- `界面结构` 用表格
- `操作顺序` 用小标题分步骤
- 避免长段落
- 控制每段长度
- 列表优先于大段说明
- 保持整篇标题风格统一
- 保持表格密度适中，避免一章里连续出现过多大表格
- 参数很多时，优先保留“第一次练习真正要用到的参数”，把其他参数压缩成补充说明

不要出现：

- 同一层级标题风格不一致
- 标题编号混乱
- 上下文跳跃
- 先讲技巧再讲主流程
- 按印象猜页面功能

## 十、内容边界

请不要写成“百科全书”。

这份文档只需要覆盖：

- 新用户第一次能走通主流程所需的信息
- 每一步最关键的界面和动作
- 依赖关系和常见误解

不需要过度展开：

- 所有参数的技术原理
- 全量 API 解释
- 所有可选能力的高级玩法
- 开发者实现细节

但有一个例外：

- 如果某个常见疑惑会直接导致新手操作失败，例如“必须先生成静态图才能生成动态参考”“按钮在滚动区域下方”“旧结果不会自动刷新”，这类信息必须写进文档。

## 十一、输出要求

请直接输出一篇完整 Markdown 文档，满足以下条件：

- 标题清晰
- 结构统一
- 适合长期维护
- 适合后续逐章迭代
- 面向第一次接触项目的新手
- 可以直接保存为 `.md`

在写作过程中，请优先保证：

1. 步骤完整
2. 顺序清楚
3. 主次分明
4. 依赖关系明确
5. 阅读体验整齐
6. 页面名称和按钮名称与真实界面一致

如果某一步是复杂页面，还请额外遵守：

1. 先用表格快速说明界面结构
2. 再用步骤带用户操作
3. 参数表只保留第一次上手最需要的内容
4. 复杂能力可以分“推荐第一次这样用”和“进阶再看”
```

## 推荐补充语句

如果你希望 AI 输出得更贴近你的习惯，可以在提示词后面追加这些限定。

### 1. 强调“不要写成功能说明书”

```md
请记住：这不是功能说明书，而是操作手册。请优先回答“用户现在该做什么”，而不是“这个模块是什么”。
```

### 2. 强调“新手优先”

```md
请以第一次接触该系统的新手为中心写作。默认读者不知道按钮含义，也不知道步骤依赖关系。
```

### 3. 强调“结构必须统一”

```md
每个步骤都必须使用同一套标题模板，不要每章临时发明新标题。
```

### 4. 强调“先操作，后解释”

```md
请先写操作顺序，再写必要解释。不要先展开讲原理，再让读者自己拼操作路径。
```

### 5. 强调“控制篇幅”

```md
请收紧语言，避免长段铺垫。不要压缩关键信息，但要主动删除重复解释。
```

### 6. 强调“源码优先，不要猜”

```md
请优先根据源码和真实页面写作。按钮名、页签名、状态名、弹窗名必须与真实界面一致；如果没有依据，请不要猜测。
```

### 7. 强调“复杂页面允许辅助表格，但不能喧宾夺主”

```md
如果某一步参数很多、模式很多，可以在该步骤内部增加少量辅助表格，但表格必须服务操作顺序，不能把整章写成参数大全。
```

### 8. 强调“区分可用功能和 Beta 边界”

```md
请不要把后续步骤一概写成“先跳过”。如果源码显示某页已经能完成真实操作，请写清楚它现在能做什么；如果只是部分可用，也请写清楚当前边界。
```

## 你可以这样给 AI 下文

下面是一段更短、更适合实际投喂的版本。

```md
请基于我提供的代码、现有文档、前端页面和实际操作流程，输出一篇“新手操作手册”。

要求：

- 这是操作手册，不是功能说明书
- 面向第一次接触项目的新手
- 重点写操作顺序、依赖关系、判断标准
- 必须优先依据源码和真实页面，不要猜
- 先告诉用户做什么，再简短解释为什么
- 每个主步骤统一使用：
  - X.0 界面结构
  - X.1 操作顺序
  - X.2 判断标准
  - X.3 关键说明
  - 完成标志
- `界面结构` 用表格
- `操作顺序` 用“步骤 1 / 步骤 2 / 步骤 3”子标题
- 页面复杂时，允许补少量服务操作的辅助表格
- 不要写成长篇纯文字
- 不要让辅助信息抢主流程篇幅
- 请主动识别哪些内容是主流程，哪些内容只是可选项或 Beta 能力
- 页面名称、按钮名称、状态名称尽量与真实界面一致
- 输出 Markdown 正文即可
```

## 最后的使用建议

如果你以后要为其他项目写类似文档，建议遵循这套顺序：

1. 先分析代码和页面，确认真实操作链路
2. 再提炼主流程和依赖关系
3. 最后才开始组织文档结构

不要反过来先写“漂亮目录”，再去补真实内容。  
真正决定一份操作手册质量的，不是辞藻，而是：

- 顺序是否清楚
- 依赖是否讲明
- 页面是否讲到点上
- 新手能不能跟着走完一遍

如果你后面已经基于真实项目打磨过一轮手册，再回头更新这份提示词时，优先补充这些内容：

- 哪些章节结构最稳，应该固化
- 哪些误区最容易反复出现，应该写进通用规则
- 哪些“看起来像细节”的信息，其实会直接影响新手能否走通流程
